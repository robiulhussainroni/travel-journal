# Architecture of Travel Journal App

I'm not going with any fancy architecture, as I already mentioned on the readme that this is a learning project, I'm trying to practice with my current knowledge level.

I created a MapManager class diagram to handle all the logic for Map. This is just the starting things may change as I progress. Reason behind creating a separate class for map is that there are so many things just happening with map. From getting the position, loading the map and also attributes like zoom level of map. When I first designed the flowchart I thought if the geoLocation isn't supported or user deny the location acces I'll show an error alert. But while start coding (just the map part, before organizing), I find that it'd be better to get the position from the user directly if he wants to open the map in different location or somehow we can't get the position. There is so many things for map that's why I thought it'd be great if I create separate class for Map.

![Class to manage Map](/docs/class-mapmanager.png)

From the start, I was keep thinking how can I create multiple classes instead of adding everything in one big App class. That's why previously I designed the MapManager class. There are few things which I believe they can be boxed inside another separate class. Everything related to travel log, that means adding new travel log, displaying the travel log in the app etc. they can be inside the Travel Manager. One challenge I faced was that I needed map related data, because without the map I couldn't really attach any handler to the map, so I created an instance of TravelManager class inside the method where the map was loading and then attached the handler on the map and as for the function I just reference the newly created instance's displayForm() method. One thing I'm realizing while architecting this app is that we will understand what our class need more once we start implementing it. I started with a rough class diagram and then ended up with more details. This class isn't fully covered yet, but the main part which is connecting this class with MapManager class is done.

![Class to manage Travel Log along with MapManager class](/docs/class-travelmanager.png)

One problem with the MapManager class was in the #getPositionDirectly() and in the #loadMap(position) there were literary simillar code, so it was violating DRY principle. One of the challenge to refactor this code was that both this method required different latitude and longitude. #loadMap(position) was calling directly from geolocation code when it successfully get the user current location and #getPositionDirectly() was the opposite, it'll call when geolocation didn't get the user current location, the goal with this method was to give user chance to render the map on the location where he wants or if somehow geoloaction isn't supported or blocked. The coordinates coming from #getPositionDirectly() was from the user input and the in the #loadMap(position) it was getting from the position argument. To solve this challenge, I had to create a separate method #getPositionGeoLocation(position), where I write all the code necessary to run successful geolocation function and get the position and the destructure the coordinates from the position. Then everything become easy I delete the duplicate code from #getPositionDirectly(), and in both the method (#getPositionGeoLocation(position), #getPositionDirectly) I called #loadMap(latitude, longitude) with their specific coordinates.

![Refactor the MapManager Class to avoid repetition](/docs/mapmanager-refactor.png)

I faced some difficulity in order to display travel details when user clicked on travelList. To solve this challenge, I try to linked the travel list and travel details. Whenever user submitting the form along with creating the list I also now pushing the code on #travelLog array and to linked travelList item and the the corresponding array elemenet I created an id. Currently, I choose a naive approach to come up with unique id which is by Date.now(). I know it's not good for production as in production many user could submit the form in the same time, then there would be bug. Currently, I'm just sticking with this as it's just learning project but maybe later I'll try to come up with different solution in order to create unique id (maybe through any library or api). In order to do this I created #travelDetails method in travel class. I'm not using any icon to remove the travel details modal when it appears after user clicking on travel list instead I'm using timer. After 5 seconds, it'll be hidden from the UI and during those 5 seconds the related travel list will be hidden from the UI. In the timer, I've also removing the innerHTML of travel modal, so when user click on new list it won't show all the details from other lists too, rather it'd just contain the current list details.

To render map marker on the click spot, I've tweaked the codebase little bit. Previously, on map click I was directly delegating the behavior to displayform() method on TravelManager class. Now, instead of doing it, I created two new methods on MapManager class which are #mapInfo() and #renderMapMarker(). On #mapInfo() I collected the Map Event and delegated the marker rendering behavior to #renderMapMarker(). Previously I was handling Map Event on TravelManager's displayform() method. But I thought it doesn't make any sense that's why I change the code and include the Map Event on MapManager class and also make it private. However, I still needed to display the form on Map Click. That's why instead of calling the TravelManager instance directly on MapManger which actually I did previously but now I changed it and create a private field on MapManger called #travel. Then I assigned a TravelManager instance to #travel private field on MapManager.

Problem A : The map marker should be rendered after the user submit the form. Now, the problem is I'm dealing with rendering map on MapManager class and dealing with form submission on TravelManager class. I had face this sitaution before when I had to call displayForm() method on MapManager class which located on TravelManager class. That time I could do it easily by creating an instance of TravelManager on the MapManager. But now when I try to create an instance for MapManager on TravalManager class even though it created but somehow it's not getting any map related value. It's really hard for me to understand why it's happening even though the map is already loaded.

Reason : The real problem is that leaflet library already initialized map L.Map() when we call it inside App class. So, when we try to create another instance inside TravelManager class it's trying to initialized the map on same element again. That's why it leads us towards 'Map container is already initialized' error.

Problem B : There is a serious bug I noticed. After first travel log, if we want to create another travel log, on every travel log there is an extra travel list item creates. On the regular travel log, the travel list displays travel place name and the date the travel log created but the extra travel list can't get any travel place name, they are being rendered with just date.

Reason : I analyzed it further and turn out on first form submit the #renderTravel() method trigger one time, on second form submit click it triggers two times, on third form submit it triggers three times and it goes on. The core problem isn't on the form submit it's on map click. Everytime I click on map, it delegates it's behavior to #mapInfo(e) method. Inside this method, we are creating an instance of TravelManager in order to display the form. So, on every new click there is an instance created for TravelManager and that's what leads to the bug.

Solution for Problem A and B : One solution that I think of and already tested it temporarily and it kind of works. I think the App class is totally useless as it's doing nothing but creating a MapManager instance. So, I thought I should remove this class and create an instance for MapManager and an instance for TravelManager outside any class. Then make #renderMapMarker() public and displayForm() is already public. So, we don't have to create a new TravelManager instance on #mapInfo() method neither we've to create a new MapManager instance on #renderTravel(e) method, we can access them from global scope. There would be no problem even though we're create those instances after the class and trying to access them inside the class as both the #mapInfo() method and #renderTravel(e) method triggers when we click on the map and submits the form.

To integrate local storage it'd be ideal to create a separate class as LocalStorageManager to manage all the code related to local storage following the patterns I'm using for TravelManager to manage code related to travel and MapManager to manage code related to map. Implementing local storage in this application won't be straight forward as far I'm concern. Three types of data I've to save on localstorage. 1. Data to render the map marker, 2. Data to render the travel list, 3. Data to display the travel details modal. As per for map marker, I'm thinking to save all the mapEvent (on map click) inside an array, as from these mapEvents we can access all the places user travels (by getting latitude and longitude). As per for 2nd and 3rd point, I've already structuring travel details inside an array.

One of the problem I faced to store mapEvent is that whenever I want to store it on localstorage using JSON.stringify() it was giving me an error of circular referece. Reason was on mapEvent there are few elements they are refering the map object itself. So, to solve this problem, I create a field 'locationDetails : []' and there I push only the latitude and longitude by destructuring mapEvent. Then I store locationDetails on localStorage and it solved the problem.

To display map marker and travel details from local storage without duplicating the code, I've to refactor renderMapMarker() and #renderTravel() methods.
I refactored the code to achieve DRY principle. During that process I've to create some extra methods. In MapManager class, I created getLatLng() method as a bridge to connect renderMapMarker. Inside getLatLng() we're getting latitude and longitude of the location and pushing the info to locationDetails feield array. Then inside renderMapMarker() we loop through locationDetails in order to avoid code duplication. Inside TravelManager class, I've updated #renderTravel(e) method. I removed the code related to rendering the list and delegated it to a new method renderList(). Simillarly, like renderMapMarker() wee loop through but this time on travelLog field array in order to follow DRY principle. LocalStorageManager class have no constructor, as we've to deal with it once the map has successfully loaded.
